评测中心让你在**部署到生产环境之前**验证智能体质量。创建一组测试用例，对任意智能体运行评测，即可获得一份包含每条用例通过/失败判定、延迟和 Token 消耗的量化报告。

<Info>
专为企业采购评审设计——每一条结果均可审计、可复现、持久存储。
</Info>

## 工作原理

每次评测运行会对每条测试用例执行 **三阶段流水线**：

```
┌──────────────────────────────────────────────────────────┐
│  阶段 1 — Agent 执行                                      │
│                                                          │
│  使用真实的 ReActAgent 运行测试用例的 prompt。              │
│  与聊天完全相同的引擎：相同模型、相同工具、相同 instructions。 │
│  无 mock、无捷径。                                        │
│                                                          │
│  → 产出：answer（回答）、latency（延迟）、token usage       │
├──────────────────────────────────────────────────────────┤
│  阶段 2 — LLM 评分                                        │
│                                                          │
│  一个独立的"评分员"LLM（fast model）对回答进行评判，         │
│  依据是 expected_behavior 和 assertions。                  │
│                                                          │
│  输入：prompt + expected_behavior + assertions + answer    │
│  输出：{ verdict: "pass"|"fail", reasoning: "..." }        │
├──────────────────────────────────────────────────────────┤
│  阶段 3 — 持久化 & 汇总                                   │
│                                                          │
│  每条用例结果写入数据库。                                   │
│  全部完成后：计算通过率、平均延迟、总 Token 并挂到 run 记录。 │
└──────────────────────────────────────────────────────────┘
```

### 关键设计决策

| 决策 | 原因 |
|------|------|
| **真实 ReActAgent**（非 mock） | 测试实际的 Agent 行为，包括工具调用和多步推理 |
| **独立评分 LLM**（fast model） | 成本低、速度快；Agent LLM 在执行阶段已经消耗了 Token |
| **`asyncio.Semaphore(5)`** | 并发上限为 5，避免打爆 LLM 提供商的速率限制 |
| **每条用例独立** | 用例之间无对话历史；每条都获得一个全新的 Agent 实例 |
| **后台执行** | 运行以异步任务启动，API 立即返回，前端每 3 秒轮询状态 |

## 使用流程

### 1. 创建数据集

进入 **评测中心 → 测试集** 标签页，点击 **新建数据集**。

数据集是一组命名的测试用例集合。取一个有意义的名称（如"客服场景——一级问题"），并可选填描述。

### 2. 添加测试用例

进入数据集后添加测试用例。每条用例包含三个字段：

| 字段 | 是否必填 | 说明 |
|------|---------|------|
| **提示词（Prompt）** | 是 | 发送给 Agent 的具体问题或指令 |
| **期望行为（Expected Behavior）** | 是 | 用自然语言描述正确回答应该是什么样的 |
| **断言（Assertions）** | 否 | 一组具体检查项（如"回答须提到退款政策"、"回复不超过 200 字"） |

<Tip>
**编写优质测试用例的建议：**
- **提示词**：尽量具体。"企业套餐的退款政策是什么？"优于"跟我说说退款"。
- **期望行为**：描述*结果*而非精确用词。例如"解释 30 天退款窗口，并提到年付例外情况"。
- **断言**：用于硬性要求。"必须提到价格"比期望评分员自行判断更可靠。
</Tip>

### 3. 开始评测

切换到 **评测记录** 标签页，点击 **新建评测**。选择：

1. **智能体** — 你拥有的任意 Agent
2. **数据集** — 包含至少一条用例的数据集

点击 **开始评测**，页面将自动跳转到结果页。

### 4. 查看结果

结果页面包含：

- **顶部信息**：Agent 名称、数据集名称、状态徽章、通过率、平均延迟、总 Token
- **进度条**：随用例完成而填充（绿色 = 通过比例）
- **结果表格**：每条用例一行，包含：
  - 提示词（截断显示——点击展开）
  - 判定：**通过**（绿色）、**未通过**（红色）、**错误**（橙色）
  - Agent 回答（截断——点击展开）
  - 评分理由（为什么通过或未通过）
  - 延迟（ms）和 Token 数

<Warning>
运行进行中（`pending` 或 `running` 状态）时，页面每 3 秒自动刷新。如需实时查看结果出现，请勿离开该页面。
</Warning>

## 测试范围

### 包含
- **内置工具**：calculator、web_search、web_fetch、python_exec、file_ops 等
- **Agent 指令**：Agent 的 `instructions` 字段会被传入
- **Agent 配置的模型**：如果 Agent 有自定义模型配置，将使用该模型

### 不包含（有意为之）
- **连接器（Connector）**：外部 HTTP 连接器依赖第三方服务——评测时跳过以避免不稳定
- **MCP 服务器**：同理——外部进程依赖
- **对话历史**：每条用例独立运行，无前置上下文
- **知识库**：KB 检索工具不在评测模式下加载

<Info>
这意味着评测结果反映的是 Agent 的**推理能力和工具使用能力**，而非它与外部服务的集成效果。连接器请通过连接器测试功能单独验证。
</Info>

## 评分员（Grader）

评分员是一个 LLM（系统的 "fast" 模型），接收四条信息并返回结构化 JSON 判定：

**系统提示：**
> You are an impartial AI evaluator. Your job is to judge whether an AI agent's answer meets the expected behavior for a given prompt.
>
> Be strict but fair. A "pass" requires the answer to genuinely address the prompt according to the expected behavior. A "fail" means the answer is wrong, incomplete, off-topic, or misses key requirements.

**用户消息包含：**
1. 原始提示词
2. 期望行为
3. 断言列表（或 "None specified"）
4. Agent 的实际回答

**输出 Schema：**
```json
{
  "verdict": "pass" | "fail",
  "reasoning": "通过或未通过的理由说明..."
}
```

评分员使用 `structured_llm_call`（function calling）来强制输出格式。如果评分员自身失败（网络错误、格式异常），该用例将标记为 `error`。

## 最佳实践

### 数据集设计
- **从小处着手**：5-10 条用例覆盖 Agent 的核心场景
- **覆盖边界情况**：至少包含 2-3 条对抗性或超范围的提示词
- **明确期望行为**：模糊的期望会导致评分不一致
- **断言用于硬性要求**："必须提及价格"比期望评分员自行判断更可靠

### 解读结果
- **80%+ 通过率** 对于配置良好的 Agent 是一个不错的基线
- **低通过率 + 高延迟** 表明 Agent 在多步推理上遇到了困难
- **错误状态** 表示 Agent 或评分员崩溃——检查服务端日志了解详情
- **对评分有异议**：如果你认为评分员判断有误，先阅读其 reasoning。你可能需要改进期望行为的描述

### 何时重新评测
- 修改 Agent 指令后
- 切换 Agent 的 LLM 模型后
- 添加或移除工具类别后
- 任何生产部署之前（CI/CD 集成将在后续版本推出）

## API 参考

| 方法 | 端点 | 说明 |
|------|------|------|
| `POST` | `/api/eval/datasets` | 创建数据集 |
| `GET` | `/api/eval/datasets` | 列出数据集（分页） |
| `GET` | `/api/eval/datasets/{id}` | 获取数据集 |
| `PUT` | `/api/eval/datasets/{id}` | 更新数据集 |
| `DELETE` | `/api/eval/datasets/{id}` | 删除数据集（级联删除用例） |
| `POST` | `/api/eval/datasets/{id}/cases` | 添加测试用例 |
| `GET` | `/api/eval/datasets/{id}/cases` | 列出用例（分页） |
| `PUT` | `/api/eval/datasets/{id}/cases/{caseId}` | 更新用例 |
| `DELETE` | `/api/eval/datasets/{id}/cases/{caseId}` | 删除用例 |
| `POST` | `/api/eval/runs` | 启动评测运行 |
| `GET` | `/api/eval/runs` | 列出运行记录（分页） |
| `GET` | `/api/eval/runs/{id}` | 获取运行 + 所有用例结果 |
| `DELETE` | `/api/eval/runs/{id}` | 删除运行（级联删除结果） |

所有端点均需 JWT 认证（`Authorization: Bearer <token>`）。
