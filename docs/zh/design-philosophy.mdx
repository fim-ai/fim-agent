---
title: "设计理念"
description: "DAG 执行设计中的架构决策——我们选择不构建什么，以及为什么。"
---

FIM One 的 DAG 执行引擎在设计时保持了刻意的克制。多个在多智能体框架中常见的模式经过评估后被有意排除。本页记录了这些决策及其背后的推理。

## Handoff（任务交接）— 不需要

**模式**：智能体之间的显式任务交接，一个智能体完成工作后将控制权传递给下一个。

**排除原因**：DAG 的线性依赖链已经天然覆盖了顺序任务交接。当步骤 B 依赖步骤 A 时，执行器会等待 A 完成，然后通过依赖机制将 A 的结果传入 B 的上下文。这在结构上等价于 Handoff——但无需引入额外的抽象层。

添加显式的 Handoff 协议只会重复步骤依赖已经提供的功能，而不增加任何表达能力。

## Backflow（回传）— 不需要

**模式**：下游步骤将结果回传给上游步骤以进行重新评估（也称为结果反向传播）。

**排除原因**：PlanAnalyzer 的重新规划循环已经覆盖了这一场景。所有步骤完成后，分析器会评估原始目标是否达成。如果未达成（且置信度低于停止阈值），它会携带上一轮结果的上下文——包括哪里出了问题以及学到了什么——触发一次完整的重新规划。

这是一种粒度更粗但更稳健的方法。逐步骤回传会创建复杂的反馈回路，难以调试且可能导致无限循环。重新规划循环以更简洁的执行模型实现了同样的纠正效果：规划、执行、评估、按需重新规划。

## Teams（智能体间通信）— 不需要

**模式**：多个专业化智能体在执行过程中相互通信，协商子任务归属，或在共享状态上协作。

**排除原因**：DAG 依赖链足以处理数据依赖型任务。当步骤 C 需要步骤 A 和步骤 B 的结果时，依赖边直接表达了这种关系。步骤 C 在其上下文中接收 A 和 B 的输出——无需智能体间消息协议。

DAG 模型比智能体间通信更简单、更可预测。智能体不需要协商；规划器预先确定执行结构，执行器通过依赖解析强制执行数据流。这避免了多智能体消息传递带来的协调开销和不确定性。

## agent_hint / role_hint（角色提示）— 不需要

**模式**：显式提示告诉执行器每个步骤应由哪种类型的智能体或角色来处理（例如"使用研究员智能体"或"使用编码智能体"）。

**排除原因**：现有的三个字段组合已经提供了充分的指导：

- **`task`** — 步骤应完成内容的自然语言描述。一个写得好的任务描述会隐式地传达所需的专业领域。
- **`tool_hint`** — 建议步骤应使用的工具（例如 `web_search`、`python_exec`）。这比模糊的角色标签更精确地约束了执行方式。
- **`model_hint`** — 控制步骤使用快速模型还是推理模型层级，这是影响最大的执行决策。

添加 `agent_hint` 或 `role_hint` 会与这三个字段产生冗余，并引入规划器 LLM 必须正确处理的又一个配置维度。保持提示面小有助于减少规划错误。
